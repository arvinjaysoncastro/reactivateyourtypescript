# 1.1 Const vs Let

Phase: JS Reactivation
Topic: Variables
Status: Expanded

## Fast Goal
Use `const` by default and understand when `let` is actually needed.

## Brain Hook
`const` protects the label. `let` lets the label move.

## Why This Matters

This is one of the first places modern JavaScript style shows up.
If you get this right early, your code becomes easier to trust and easier to reason about.

The key distinction is this:

- `const` stops reassignment of the variable itself.
- `let` allows reassignment.
- Neither keyword magically deep-freezes objects.

## Micro Rule

Use `const` by default.
Switch to `let` only when the variable must point to a new value later.

## Core Idea

`const` means the binding cannot be reassigned.

```ts
const score = 10;
// score = 20; // Error
```

`let` means the binding can be reassigned.

```ts
let score = 10;
score = 20;
```

With objects, `const` protects the variable binding, not the inside of the object.

```ts
const user = { name: "Ava" };
user.name = "Liam";
```

That works because the object is still the same object. Only one property changed.

This does not work:

```ts
const user = { name: "Ava" };
// user = { name: "Liam" }; // Error
```

That fails because now you are trying to reassign the whole variable.

## Fast Comparison

### Reassignment

```ts
let count = 1;
count = 2;
```

Valid because `let` allows the variable to point somewhere else.

### Mutation

```ts
const settings = { theme: "light" };
settings.theme = "dark";
```

Valid because the variable still points to the same object.

### Invalid Reassignment

```ts
const settings = { theme: "light" };
// settings = { theme: "dark" }; // Error
```

Invalid because the binding itself is being changed.

## Mental Model

Think of the variable as a sticky note attached to a box.

- `const`: the sticky note must stay on the same box
- `let`: the sticky note can move to a different box
- object mutation: changing something inside the box

## Tiny Example

```ts
const user = { name: "Ava", age: 30 };
user.age = 31;

let status = "offline";
status = "online";

console.log(user);
console.log(status);
```

Expected output:

```ts
{ name: "Ava", age: 31 }
"online"
```

## Common Mistakes

- Using `let` for everything out of habit.
- Thinking `const` means fully immutable.
- Confusing property mutation with variable reassignment.
- Using `console.log()` instead of a real return or state change when the task asks for data movement.

## Debug Check

What is wrong with this code?

```ts
const count = 1;
count = 2;
```

Answer:

`count` was declared with `const`, so the binding cannot be reassigned.

## Meeting Answer

If someone asks, "Why do most teams prefer const by default?", a strong short answer is:

`const` reduces accidental reassignment, makes intent clearer, and makes code easier to reason about. We switch to `let` only when the variable genuinely needs to change.

## Done Signal

You are done with this step when you can say all three clearly:

- `const` blocks reassignment
- `let` allows reassignment
- `const` objects can still have mutated properties
