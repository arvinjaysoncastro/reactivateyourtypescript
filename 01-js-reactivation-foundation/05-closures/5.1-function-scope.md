# 5.1 Function Scope

Phase: JS Reactivation
Topic: Closures
Status: Expanded

## Fast Goal
- Know where variables are accessible and why that matters for closures.

## Brain Hook
- Variables belong to the function that declares them — inner functions can see outer ones, outer can't see inner.

## Why This Matters
- Predicting variable visibility prevents bugs like accidental global writes and helps you build correct closures for stateful logic.

## Micro Rule
- A variable is accessible only inside its declaring function and any nested functions (unless global).

## Core Idea
- Functions create scope. Inner functions close over variables from outer scopes, forming closures that keep access to those variables even after the outer function returns.

## Code Examples
- Good (closure capturing):
```ts
function makeCounter() {
	let count = 0;
	return () => ++count;
}
const inc = makeCounter();
inc(); // 1
inc(); // 2
```

- Bad (accidentally global):
```ts
function bad() {
	x = 5; // creates global `x` in non-strict JS — avoid
}
```

## Fast Comparison
- `var` vs `let/const`: `var` is function-scoped and can leak; `let/const` are block-scoped and safer.

## Mental Model
- Think of each function as a room: inner rooms can see items in outer rooms; outer rooms cannot see inside inner rooms.

## Tiny Example
- `const x = 1; function f(){ const y = 2; return x + y }` — `f()` can access both `x` and `y`.

## Output Prediction Drill
- What prints?
```ts
function outer(){
	const a = 'A';
	function inner(){
		console.log(a);
	}
	return inner;
}
const fn = outer();
fn();
```
- Answer: prints `'A'` — `inner` retains access to `a` via closure.

## Common Mistakes
- Trying to access inner variables from outside the function.
- Using `var` and unintentionally creating function-scoped leaks.
- Returning functions that mutate shared state without clear intent.

## Debug Check
- Broken code:
```ts
function f(){
	for(var i=0;i<3;i++){
		setTimeout(()=>console.log(i),0);
	}
}
f();
```
- What's wrong? Answer: `var` is function-scoped so all timeouts see the final `i` value (3). Use `let` or capture `i` in a new scope.

## Meeting Answer
- "Function scope means inner functions can access outer variables; closures keep those variables alive."

## Perfect Explanation
- Functions create scopes; closures allow inner functions to access variables from outer scopes after the outer returns.
