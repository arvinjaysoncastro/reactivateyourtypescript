# 5.2 Inner Function Memory

Phase: JS Reactivation
Topic: Closures
Status: Expanded

## Fast Goal
- Observe how inner functions retain access to outer variables even after outer returns.

## Brain Hook
- A closure is the memory that travels with a function.

## Why This Matters
- Closures enable private state, factory functions, and safe encapsulation — common in event handlers and libraries.

## Micro Rule
- If a function references an outer variable, that variable stays alive for as long as the inner function exists.

## Core Idea
- When a function returns an inner function that references its variables, those variables are kept in a closure and remain accessible to the inner function later.

## Code Examples
- Memory example:
```ts
function maker(){
	let secret = 1;
	return () => secret++;
}
const next = maker();
next(); // 1
next(); // 2
```

- Bad (losing state):
```ts
function bad(){
	let s = 0;
	setTimeout(()=>{ s = 5 }, 0);
	return () => s; // timing/ordering issues if caller expects immediate change
}
```

## Fast Comparison
- Closures vs globals: closures keep state private; globals are shared and error-prone.

## Mental Model
- Think of a closure as a backpack: the returned function carries needed variables with it.

## Tiny Example
- `const f = (()=>{let x=0; return ()=>++x})()` => `f()` increments a private `x`.

## Output Prediction Drill
- What prints?
```ts
function outer(){
	let n = 1;
	return () => console.log(n++);
}
const fn = outer();
fn();
fn();
```
- Answer: prints `1` then `2` — `fn` remembers `n`.

## Common Mistakes
- Assuming the outer scope variable is reset each call instead of preserved.
- Relying on globals instead of closures for private state.
- Creating unintended shared state between instances.

## Debug Check
- Broken code:
```ts
function f(){
	for(var i=0;i<3;i++){
		setTimeout(()=>console.log(i), 0);
	}
}
f();
```
- What's wrong? Answer: `var` is function-scoped; all callbacks see the final `i` (3). Use `let` or capture `i`.

## Meeting Answer

## Perfect Explanation
