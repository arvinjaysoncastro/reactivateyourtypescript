# 6.4 Simple Error Handling

Phase: JS Reactivation
Topic: Async
Status: Expanded

## Fast Goal
- Handle promise failures and provide safe fallbacks.

## Brain Hook
- Uncaught async errors become invisible bugs in production.

## Why This Matters
- Prevents crashes and preserves user experience when remote calls fail.
- In real code, catching async errors avoids data loss and noisy logs.

## Micro Rule
- Always catch promises you depend on; prefer explicit fallbacks.

## Core Idea
- Async operations can reject; you must handle rejections with `.catch()` or `try/catch` around `await` to provide safe behavior.

## Code Examples
- Good (catch and fallback):
```ts
fetchData()
	.then(data => show(data))
	.catch(() => showFallback())
```

- Good with async/await:
```ts
try {
	const data = await fetchData()
	show(data)
} catch (err) {
	showFallback()
}
```

- Bad (no handling):
```ts
await fetchData()
// no catch -> unhandled rejection
```

## Fast Comparison
- `.catch()` chains vs `try/catch` with `await`: both handle failures; `try/catch` is clearer for multi-step async code.

## Mental Model
- Think of `catch` as a safety net below a trapeze act — if the promise falls, the net catches it.

## Tiny Example
```ts
const p = Promise.reject(new Error('fail'))
p.catch(() => console.log('handled'))
```

## Output Prediction Drill
What prints?
```ts
const p = Promise.reject('boom')
p.catch(x => console.log('caught', x))
```
Answer: prints `caught boom`.

## Expected Solution
- `safeFetch` returns remote JSON when ok; otherwise returns a deterministic fallback object and does not throw.

## Common Mistakes
- Forgetting to `return` after `catch`.
- Swallowing errors silently without a fallback.
- Using a catch that masks all errors (including programmer errors).

## Debug Check
What is wrong?
```ts
async function load() {
	try {
		const res = await fetch('/api')
		res.json()
	} catch (e) {
		return { status: 'fallback' }
	}
}
```
Answer: `res.json()` is not returned; should `return res.json()` inside the `try`.
