# 7.4 Print Result

Phase: JS Reactivation
Topic: Build #1 - User Processor
Status: Expanded

## Fast Goal
- Run the final pipeline and present a clear, testable output.

## Brain Hook
- If you can't inspect the output, you can't trust the code you wrote.

## Why This Matters
- Verifying the final output ensures the pipeline produced the expected shape and values before shipping or further processing.

## Micro Rule
- Always log or return a deterministic result shape from the pipeline so tests and humans can verify it.

## Core Idea
- Combine filtering and mapping into a final `processor` function that returns the final value; prefer returning over console-only side effects for easier testing.

## Code Examples
- Good (return result):
```ts
function processor(users) {
	return users.filter(u => u.age >= 18).map(u => `${u.name} (${u.age})`)
}
const result = processor(users)
```

- Bad (only console):
```ts
function processor(users) {
	users.filter(...).map(...).forEach(x => console.log(x))
}
// hard to test; returns undefined
```

## Fast Comparison
- Prefer returning data for tests and logging for quick debugging; combine both when useful but keep return as source-of-truth.

## Mental Model
- Think of the processor as a factory: input goes in, boxed output comes out ready for inspection.

## Tiny Example
```ts
const out = [{id:1,name:'A'}].map(u => u.name)
// out -> ['A']
```

## Output Prediction Drill
What prints and what is `result`?
```ts
function processor(users){
	return users.filter(u => u.active).map(u => u.email)
}
const result = processor([{id:1,active:true,email:'a@example'},{id:2,active:false,email:'b@x'}])
console.log(result)
```
Answer: prints `['a@example']`; `result` is `['a@example']`.

## Practice

### Drill 2
- Add a step that limits results to the first 3 entries.

### Drill 3
- Return an object `{ count, items }` where `count` is number of results and `items` is the mapped list.

### Drill 4
- Explain in 1–2 lines why returning `{count, items}` helps testing.

## Your Task
- Implement `processor(users)` that filters adults, maps to `Name (age)`, and returns `{ count, items }`.

## Expected Solution
- `const items = users.filter(...).map(...); return { count: items.length, items }`

## Common Mistakes
- Only `console.log`ing without returning.
- Mutating `users` during processing.
- Not handling empty or missing arrays.

## Debug Check
What is wrong?
```ts
function processor(users) {
	const items = users.filter(u => u.age >= 18).map(u => `${u.name} (${u.age})`)
	console.log(items)
}
```
Answer: function logs but does not return `items`, making it hard to test or reuse.

## Meeting Answer
- "My processor returns a deterministic result object `{count, items}` so we can assert correctness and inspect output easily."

## Perfect Explanation
- Return a deterministic result shape from the pipeline so tests can verify it and humans can inspect it.

## Guiding Questions
- Should the processor `return` or `console.log` the result?
- How should empty inputs be represented?
- Do we need to trim or normalize fields before mapping?

## Done Signal
- `processor` returns `{ count, items }` with expected items.
- Works with empty arrays and missing fields documented.
- No mutation of the original `users` array.
