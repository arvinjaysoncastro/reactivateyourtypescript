# 5.3 Counter Pattern

Phase: JS Reactivation
Topic: Closures
Status: Expanded

## Fast Goal
- Build a simple stateful closure (counter) and explain its private state.

## Brain Hook
- The counter is the smallest useful closure: private state + controlled access.

## Why This Matters
- Counters demonstrate how to keep state private, avoid globals, and expose controlled behavior — patterns used in components, hooks, and libraries.

## Micro Rule
- Keep mutable state inside the factory and expose only functions that operate on it.

## Core Idea
- A counter factory creates a private `count` variable and returns functions that read or modify it; returned functions close over `count` so it persists.

## Code Examples
- Good:
```ts
function makeCounter(initial = 0) {
	let count = initial;
	return {
		inc: () => ++count,
		dec: () => --count,
		get: () => count,
		reset: () => (count = initial),
	};
}
```

- Bad (global):
```ts
let count = 0; // global — shared and error-prone
function inc(){ return ++count }
```

## Fast Comparison
- Counter closure vs global: closure gives instance-specific private state; global shares state across the app.

## Mental Model
- Think of each counter as an independent battery: its charge (`count`) is hidden inside and only accessible through terminals (methods).

## Tiny Example
- `const c = makeCounter(); c.inc(); c.get(); // 1`

## Output Prediction Drill
- What prints?
```ts
const c1 = makeCounter(1);
const c2 = makeCounter(10);
```

## Common Mistakes
- Creating `count` in the global scope.
- Returning functions that mutate shared external state.
- Forgetting to return the updated value from `inc`/`dec`.

## Debug Check
- Broken code:
```ts
function factory(){
	for(var i=0;i<3;i++){
		// returns functions that all reference same `i`
	}
}
```
- What's wrong? Answer: `var` is function-scoped; closures capture the final `i`. Use `let` or create a fresh scope per iteration.
