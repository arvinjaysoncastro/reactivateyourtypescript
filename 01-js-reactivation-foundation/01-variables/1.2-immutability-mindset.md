# 1.2 Immutability Mindset

Phase: JS Reactivation
Topic: Variables
Status: Expanded

## Fast Goal
Think in safe updates instead of accidental mutation.

## Brain Hook
Do not change the box if you can create a better box.

## Why This Matters

In real code, data often gets shared across functions, components, and services.
If one part of the code mutates that shared data directly, another part can start behaving differently without warning.

The goal is not to avoid every mutation forever.
The goal is to build the habit of making updates in a way that is easier to trust, trace, and debug.

## Micro Rule

When you need updated data, prefer creating a new object or array instead of changing the original one directly.

## Core Idea

Immutability means you treat existing data as read-only and create an updated copy when something needs to change.

```ts
const user = { name: "Ava", active: false };

const updatedUser = {
	...user,
	active: true,
};
```

In that example, `user` stays unchanged.
`updatedUser` is the new value with the change applied.

This is the mutating version:

```ts
const user = { name: "Ava", active: false };
user.active = true;
```

That works, but now the original object was changed in place.
If other code was relying on the old value, you just changed shared state directly.

## Fast Comparison

### Mutating Update

```ts
const profile = { name: "Ava", city: "Cebu" };
profile.city = "Manila";
```

The original `profile` object is changed.

### Copy-Based Object Update

```ts
const profile = { name: "Ava", city: "Cebu" };

const updatedProfile = {
	...profile,
	city: "Manila",
};
```

The original object stays the same.
The new object contains the update.

### Copy-Based Array Update

```ts
const skills = ["js", "ts"];
const updatedSkills = [...skills, "react"];
```

The original array stays unchanged.

## Mental Model

Think of your data like a saved document.

- mutation: overwrite the original file
- immutability: save a new version with the change

That second approach is often safer because you can still inspect the old version if something goes wrong.

## Tiny Example

```ts
const settings = {
	theme: "light",
	notifications: true,
};

const updatedSettings = {
	...settings,
	theme: "dark",
};

console.log(settings);
console.log(updatedSettings);
```

Expected output:

```ts
{ theme: "light", notifications: true }
{ theme: "dark", notifications: true }
```

## Common Mistakes

- Updating the original object directly and calling it immutable.
- Using `.push()` on the original array instead of creating a new array.
- Forgetting that immutability is about the update strategy, not about whether JavaScript technically allows mutation.
- Copying the object but forgetting to apply the changed property in the new object.

## Debug Check

What is wrong with this code?

```ts
const user = { name: "Mika", active: false };
const updatedUser = user;
updatedUser.active = true;
```

Answer:

`updatedUser` is not a new object. It points to the same object as `user`, so changing `updatedUser.active` also mutates `user`.

## Meeting Answer

If someone asks, "Why do teams care about immutability?", a strong short answer is:

`Immutability makes state changes easier to predict, debug, and review because we create updated copies instead of silently changing shared data in place.`

## Done Signal

You are done with this step when you can say all three clearly:

- immutability means prefer updated copies
- direct mutation changes the original value in place
- copy-based updates make behavior easier to trace
