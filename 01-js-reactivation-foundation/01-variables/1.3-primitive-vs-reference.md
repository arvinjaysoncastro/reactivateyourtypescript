# 1.3 Primitive vs Reference

Phase: JS Reactivation
Topic: Variables
Status: Expanded

## Fast Goal
Know what gets copied by value and what shares a reference.

## Brain Hook
Primitive copies the value. Objects and arrays copy the address.

## Why This Matters

This is one of the main reasons JavaScript data behavior can feel confusing at first.
If you do not know whether a value was copied or shared, you can misread output, introduce bugs, or accidentally change data in another part of the code.

Once this distinction is clear, mutation, immutability, and debugging all become easier.

## Micro Rule

Primitives copy the value itself.
Objects and arrays copy a reference to the same underlying data unless you create a new copy on purpose.

## Core Idea

Primitive values like strings, numbers, and booleans are copied by value.

```ts
let a = 10;
let b = a;

b = 20;
```

Now `a` is still `10` and `b` is `20` because `b` got its own copy of the value.

Objects and arrays behave differently.

```ts
const user1 = { name: "Ava" };
const user2 = user1;

user2.name = "Liam";
```

Now both `user1.name` and `user2.name` are `"Liam"` because both variables point to the same object.

## Fast Comparison

### Primitive Copy

```ts
let city1 = "Cebu";
let city2 = city1;

city2 = "Manila";
```

`city1` stays `"Cebu"`.
`city2` becomes `"Manila"`.

### Reference Copy

```ts
const settings1 = { theme: "light" };
const settings2 = settings1;

settings2.theme = "dark";
```

Both variables now show `theme: "dark"` because they share the same object reference.

### Safe Reference Update

```ts
const settings1 = { theme: "light" };
const settings2 = { ...settings1, theme: "dark" };
```

Now the two variables point to different objects.

## Mental Model

Think of a primitive as a copied note.
Think of an object or array as a shared map location.

- primitive copy: you get your own note
- reference copy: both variables point to the same place
- copied object: you create a new place with similar contents

## Tiny Example

```ts
let count1 = 1;
let count2 = count1;
count2 = 2;

const tags1 = ["js"];
const tags2 = tags1;
tags2.push("ts");

console.log(count1, count2);
console.log(tags1, tags2);
```

Expected output:

```ts
1 2
["js", "ts"] ["js", "ts"]
```

## Expected Solution

```ts
let score1 = 10;
let score2 = score1;
score2 = 20;

const player1 = { name: "Ava" };
const player2 = player1;
player2.name = "Liam";

console.log(score1);
console.log(score2);
console.log(player1);
console.log(player2);

// Primitives copy the value, but objects copy a shared reference.
```

## Common Mistakes

- Thinking every assignment creates a full independent copy.
- Forgetting that arrays are reference values too.
- Confusing reassignment of a variable with mutation of a shared object.
- Assuming `const` makes an object independent from another reference.

## Debug Check

What is wrong with this reasoning?

```ts
const a = { level: 1 };
const b = a;
b.level = 2;

// "Only b changed because I updated b."
```

Answer:

That reasoning is wrong because `a` and `b` point to the same object. Updating `b.level` mutates the shared object, so `a.level` also becomes `2`.

## Meeting Answer

If someone asks, "What is the difference between primitive and reference assignment?", a strong short answer is:

`Primitives copy the value itself, while objects and arrays copy a reference to the same underlying data. That is why object updates can affect both variables unless you create a new copy.`
